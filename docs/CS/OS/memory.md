## 简介

进程的地址空间包含运行程序的所有内存状态。

内存层次结构

静态 RAM 缓存
DRAM 主内存帧缓冲区

操作系统中管理（部分）内存层次结构的部分称为内存管理器。
其工作是高效管理内存：跟踪哪些部分的内存正在使用，在进程需要时分配内存，并在进程完成时释放内存。

> [!NOTE]
>
> Every address you see is virtual, only the OS (and the hardware) knows the real truth.

### 目标

- 虚拟内存系统的一个主要目标是透明性。操作系统应以对运行程序不可见的方式实现虚拟内存。
- 虚拟内存的另一个目标是效率。操作系统应努力使虚拟化尽可能高效，无论是在时间还是空间方面。
- 最后，虚拟内存的第三个目标是保护。操作系统应确保保护进程免受彼此影响，以及保护操作系统自身免受进程影响。

向进程暴露物理内存有几个主要缺点。

- 首先，如果用户程序可以寻址内存的每个字节，它们可以有意或意外地轻易破坏操作系统，导致系统瘫痪。
  即使只运行一个用户程序（应用程序），此问题也存在。其次，使用此模型时，难以同时运行多个程序。
- 在个人计算机上，同时打开多个程序是常见的，其中一个具有当前焦点，但其他程序在点击鼠标时会被重新激活。
  由于在没有物理内存抽象的情况下难以实现这一点，因此必须采取某些措施。

必须解决两个问题才能允许多个应用程序同时驻留在内存中而不相互干扰：保护和重定位。

### 动态（基于硬件的）重定位

一个更好的解决方案是为内存发明一种新的抽象：地址空间。
正如进程概念创建了一种抽象 CPU 来运行程序一样，地址空间创建了一种抽象内存供程序驻留。
地址空间是进程可用于寻址内存的地址集合。
每个进程都有自己的地址空间，独立于其他进程的地址空间（除非在某些特殊情况下进程想要共享其地址空间）。

具体来说，我们需要每个 CPU 内的两个硬件寄存器：一个称为基址寄存器，另一个称为界限寄存器（有时称为限制寄存器）。
这对基址-界限寄存器允许我们将地址空间放置在物理内存中任何我们希望的位置，同时确保进程只能访问其自己的地址空间。

在此设置中，每个程序都像加载在地址零处一样编写和编译。
然而，当程序开始运行时，操作系统决定它应在物理内存中的何处加载，并将基址寄存器设置为该值。
在上面的例子中，操作系统决定将进程加载到物理地址 32 KB 处，因此将基址寄存器设置为该值。

当进程运行时，有趣的事情开始发生。
现在，当进程生成任何内存引用时，处理器按以下方式对其进行转换：

```
physical address = virtual address + base
```

> The basic technique is referred to as static relocation, in which a piece of software known as the loader takes an executable that is about to be run and rewrites its addresses to the desired offset in physical memory.
>
> However, static relocation has numerous problems.
>
> - First and most importantly, it does not provide protection, as processes can generate bad addresses and thus illegally access other process's or even OS memory
> - Another negative is that once placed, it is difficult to later relocate an address space to another location.

进程生成的每个内存引用都是**虚拟地址**；硬件然后将基址寄存器的内容添加到此地址，结果是可发送到内存系统的**物理地址**。

将虚拟地址转换为物理地址正是我们称为**地址转换**的技术；也就是说，硬件获取进程认为它正在引用的虚拟地址，并将其转换为数据实际所在的物理地址。
因为这种地址重定位发生在运行时，并且因为即使在进程开始运行后我们也可以移动地址空间，所以这种技术通常称为**动态重定位**。

界限（或限制）寄存器确保此类地址在地址空间的范围内。

我们应该注意，基址和界限寄存器是芯片上的硬件结构（每个 CPU 一对）。
有时人们将帮助进行地址转换的处理器部分称为**内存管理单元**（**MMU**）；随着我们开发更复杂的内存管理技术，我们将在 MMU 中添加更多电路。

操作系统需求

- 内存管理
  - 需要为新进程分配内存；
  - 从终止的进程回收内存；
  - 通常通过空闲列表管理内存
- 基址/界限管理 在上下文切换时必须正确设置基址/界限
- 异常处理 异常发生时要运行的代码；可能的操作是终止违规进程

## 分页

不将进程的地址空间分割为若干可变大小的逻辑段（例如代码、堆、栈），而是将其分割为*固定大小的单元*，每个单元称为一个页。

大多数虚拟内存系统使用称为分页的技术。
地址可以使用索引、基址寄存器、段寄存器等方式生成。

这些程序生成的地址称为虚拟地址，构成虚拟地址空间。
在没有虚拟内存的计算机上，虚拟地址直接放到内存总线上，导致具有相同地址的物理内存字被读取或写入。
当使用虚拟内存时，虚拟地址不会直接进入内存总线。
相反，它们进入 MMU（内存管理单元），将虚拟地址映射到物理内存地址。

虚拟地址空间由称为页的固定大小单元组成。物理内存中对应的单元称为页帧。页和页帧通常大小相同。

在实际硬件中，*存在/缺失位*跟踪哪些页实际在物理内存中。
如果程序引用了未映射的地址，MMU 会注意到该页未映射（在图中用叉号表示），并导致 CPU 陷入操作系统。
这种陷阱称为*缺页中断*。
操作系统选择一个不常用的页帧，将其内容写回磁盘（如果尚未在磁盘上）。
然后它从磁盘取出刚引用的页到刚释放的页帧中，修改映射，并重新启动被陷阱的指令。

### 页表

为了记录地址空间的每个虚拟页在物理内存中的位置，操作系统通常维护一个每进程的数据结构，称为**页表**。

在简单的实现中，虚拟地址到物理地址的映射可以总结如下：虚拟地址被分割为虚拟页号（高位）和偏移量（低位）。
例如，对于 16 位地址和 4 KB 页大小，高 4 位可以指定 16 个虚拟页中的一个，低 12 位则指定所选页内的字节偏移量（0 到 4095）。
然而，使用 3 位、5 位或其他位数的分割也是可能的。不同的分割意味着不同的页大小。

虚拟页号用作页表的索引，以查找该虚拟页的条目。从页表条目中，找到页帧号（如果有）。
页帧号附加到偏移量的高端，替换虚拟页号，形成可发送到内存的物理地址。

因此，页表的目的是将虚拟页映射到页帧。
从数学上讲，页表是一个函数，以虚拟页号为参数，物理帧号为结果。
使用此函数的结果，虚拟地址中的虚拟页字段可以被页帧字段替换，从而形成物理内存地址。

#### 页表条目

最简单的形式称为线性页表，它只是一个数组。

大小因计算机而异，但 32 位是常见大小。最重要的字段是页帧号。
毕竟，页映射的目标是输出这个值。旁边是存在/缺失位。如果该位为 1，则条目有效且可用。
如果为 0，则该条目所属的虚拟页当前不在内存中。访问存在位为 0 的页表条目会导致缺页中断。

*保护位*指示允许哪些类型的访问。在最简单的形式中，此字段包含 1 位，0 表示读/写，1 表示只读。
更复杂的安排是使用 3 位，分别用于启用页的读取、写入和执行。

*修改位*和*引用位*跟踪页的使用情况。当页被写入时，硬件自动设置修改位。
当操作系统决定回收页帧时，该位很有用。如果页帧中的页已被修改（即"脏"），必须将其写回磁盘。
如果未被修改（即"干净"），可以直接丢弃，因为磁盘副本仍然有效。
该位有时称为*脏位*，因为它反映了页的状态。

*引用位*在页被引用（无论读取还是写入）时设置。其值用于帮助操作系统在发生缺页中断时选择要驱逐的页。
未被使用的页远比正在使用的页更适合作为候选，该位在几种[页置换算法](/docs/CS/OS/memory.md)中发挥着重要作用。

最后，最后一位允许禁用页的缓存。此功能对于映射到设备寄存器而非内存的页很重要。
如果操作系统在紧密循环中等待某个 I/O 设备响应刚刚发出的命令，硬件必须持续从设备获取数据，而不能使用旧的缓存副本。
通过此位，可以关闭缓存。具有独立 I/O 空间且不使用内存映射 I/O 的机器不需要此位。

请注意，用于在页不在内存时保存页的磁盘地址不是页表的一部分。原因很简单。
页表只包含硬件将虚拟地址转换为物理地址所需的信息。
操作系统处理缺页所需的信息保存在操作系统内部的软件表中。硬件不需要它。

虚拟内存从根本上所做的是创建一种新的抽象——地址空间，它是物理内存的一种抽象，就像进程是物理处理器（CPU）的一种抽象一样。
虚拟内存可以通过将虚拟地址空间分割成页，并将每个页映射到物理内存的某个页帧或使其（暂时）未映射来实现。

在任何分页系统中，必须面对两个主要问题：

1. 从虚拟地址到物理地址的映射必须快速。
2. 如果虚拟地址空间很大，页表将会很大。

第一点是每次内存引用都必须完成虚拟到物理映射的结果。
所有指令最终都必须来自内存，许多指令也引用内存中的操作数。
因此，每条指令需要进行一次、两次或更多次页表引用。

第二点源于所有现代计算机使用至少 32 位的虚拟地址，而 64 位正成为台式机和笔记本电脑的常态。
例如，使用 4 KB 页大小时，32 位地址空间有 100 万个页，而 64 位地址空间的数量则超乎想象。

##### 多级页表

一种选择是多级页表，许多操作系统（包括 Linux）都在使用。

多级页表方法的秘诀是避免始终将所有页表保留在内存中。

##### 反向页表

分页层次结构中不断增加级别的另一种替代方案称为*反向页表*。
在这种设计中，每个物理内存中的页帧有一个条目，而不是每个虚拟地址空间的页有一个条目。

虽然反向页表节省了大量空间（至少在虚拟地址空间远大于物理内存时），但它们有一个严重的缺点：虚拟到物理的转换变得困难得多。
当进程 n 引用虚拟页 p 时，硬件不能再通过使用 p 作为页表的索引来找到物理页。相反，它必须搜索整个反向页表以查找条目 (n, p)。
此外，这种搜索必须在每次内存引用时进行，而不仅仅是在缺页时。在每次内存引用时搜索 256K 表并不是让机器飞快的办法。

解决此困境的方法是使用 TLB。如果 TLB 可以容纳所有频繁使用的页，转换可以像使用常规页表一样快。
然而，在 TLB 未命中时，必须在软件中搜索反向页表。
实现这种搜索的一种可行方法是使用虚拟地址的哈希表。当前在内存中具有相同哈希值的所有虚拟页被链接在一起。
如果哈希表的槽数与机器的物理页数相同，则平均链长只有一个条目，大大加快了映射速度。
一旦找到页帧号，新的（虚拟、物理）对被输入 TLB。

### 分页也太慢

分页相对于以前的方法（如分段）有许多优点。

- 首先，它不会导致外部碎片，因为分页（通过设计）将内存划分为固定大小的单元。
- 其次，它非常灵活，支持虚拟地址空间的稀疏使用。

然而，如果不谨慎地实现分页支持，会导致机器变慢（因为访问页表需要额外内存访问）以及内存浪费（因为内存被页表而非有用的应用程序数据填满）。

## TLB

为了加速地址转换，我们将添加一个称为（由于历史原因）转换后备缓冲器（TLB）的东西。
TLB 是芯片内存管理单元（MMU）的一部分，只是一个流行的虚拟到物理地址转换的硬件缓存；因此，更好的名称是地址转换缓存。
在每次虚拟内存引用时，硬件首先检查 TLB 以查看所需转换是否在其中；如果是，则（快速地）执行转换，无需查询页表（它包含所有转换）。

假设页遍历没有在进程的页表中找到该页，因此程序发生缺页中断。有三种可能性。

- 首先，该页可能实际上在内存中，但不在该进程的页表中。例如，该页可能已被另一个进程从磁盘调入。
  在这种情况下，我们不需要再次访问磁盘，而只需在页表中适当地映射该页。
  这是一种相当轻微的缺失，称为*轻微缺页*。
- 其次，如果需要从磁盘调入页，则发生*严重缺页*。
- 第三，程序可能只是访问了无效地址，根本不需要在 TLB 中添加映射。
  在这种情况下，操作系统通常以*段错误*终止程序。只有在这种情况下程序才做了错误的事情。
  所有其他情况都由硬件和/或操作系统自动修复——代价是一些性能损失。

> TLB valid bit != page table valid bit

在页表中，当页表条目（PTE）被标记为无效时，意味着该页尚未被进程分配，正确运行的程序不应访问它。
访问无效页时的通常响应是陷入操作系统，操作系统将通过终止进程来响应。

相反，TLB 有效位仅指 TLB 条目中是否包含有效的转换。
例如，当系统启动时，每个 TLB 条目的常见初始状态被设置为无效，因为其中尚未缓存任何地址转换。
一旦虚拟内存被启用，一旦程序开始运行并访问其虚拟地址空间，TLB 逐渐被填充，因此有效条目很快填满 TLB。

TLB 有效位在上下文切换时也很有用，我们将在下面进一步讨论。
通过将所有 TLB 条目设置为无效，系统可以确保即将运行的进程不会意外使用前一个进程的虚拟到物理转换。

### 置换策略

一种常见的方法是驱逐最近最少使用（LRU）的条目。

另一种典型方法是使用随机策略，随机驱逐 TLB 映射。

> [!TIP]
>
> RAM isn't always RAM.
>
> Sometimes randomly accessing your address space, particularly if the number of pages accessed exceeds the TLB coverage, can lead to severe performance penalties.

然而，TLB 并非对所有程序都是万能药。

特别是，如果程序在短时间内访问的页数超过 TLB 能容纳的页数，程序将产生大量的 TLB 未命中，从而运行得慢得多。
我们将这种现象称为超过 TLB 覆盖范围，对于某些程序来说，这可能是一个相当大的问题。
一种解决方案是支持更大的页大小；通过将关键数据结构映射到由更大页映射的程序地址空间区域，可以增加 TLB 的有效覆盖范围。
大页支持通常由数据库管理系统（DBMS）等程序利用，这些程序具有既大又随机访问的特定数据结构。

另一个值得提及的 TLB 问题：TLB 访问很容易成为 CPU 流水线的瓶颈，特别是在使用所谓物理索引缓存的情况下。
使用这种缓存时，必须在访问缓存之前进行地址转换，这可能会大大降低速度。
由于这个潜在问题，人们研究各种巧妙的方法来使用虚拟地址访问缓存，从而在缓存命中时避免昂贵的转换步骤。
这种所谓的虚拟索引缓存解决了一些性能问题，但也给硬件设计带来了新的问题。

## 管理空闲内存

当动态分配内存时，操作系统必须对其进行管理。
一般来说，有两种跟踪内存使用的方法：位图和空闲列表。

## 交换

多年来，发展了两种处理内存过载的通用方法。
最简单的策略称为交换，包括将每个进程完整地调入内存，运行一段时间，然后将其放回磁盘。
空闲进程主要存储在磁盘上，因此当它们不运行时不会占用任何内存（尽管其中一些会定期唤醒执行其工作，然后再次进入睡眠）。
另一种策略称为虚拟内存，允许程序即使仅部分在主内存中也能运行。

当交换在内存中产生多个空洞时，可以通过将所有进程尽可能向下移动来将它们合并为一个大的空洞。
这种技术称为内存紧缩。通常不这样做，因为它需要大量 CPU 时间。

如果预期大多数进程在运行时会增长，那么在交换进程进入或移动时分配一点额外内存可能是个好主意，
以减少移动或交换不再适合其分配内存的进程所带来的开销。
然而，当将进程交换到磁盘时，只应交换实际使用的内存；交换额外内存是浪费。

如果进程可以有两个增长段——例如，用作堆的数据段（用于动态分配和释放的变量）和用于普通局部变量及返回地址的栈段——则另一种安排自然产生。
在此图中，我们看到每个进程在分配内存的顶部有一个向下增长的栈，以及在程序文本上方有一个向上增长的数据段。
它们之间的内存可以用于任一段。
如果空间用尽，进程必须被移到一个有足够空间的空洞中，被换出内存直到可以创建足够大的空洞，或被终止。

### 交换空间

我们需要做的第一件事是在磁盘上保留一些空间用于来回移动页面。
在操作系统中，我们通常将此类空间称为交换空间，因为我们将页从内存交换出去到此空间，并将页从此空间交换到内存。

交换空间的大小很重要，因为它最终决定了系统在给定时间可以使用的最大内存页数。

## 页置换算法

当发生缺页中断时，操作系统必须选择一个页来驱逐（从内存中移除）以为新进入的页腾出空间。
如果被移除的页在内存中已被修改，则必须将其重写到磁盘以使磁盘副本保持最新。
然而，如果页未被更改（例如，它包含程序文本），磁盘副本已经是最新的，因此无需重写。
要读入的页直接覆盖被驱逐的页。

虽然在每次缺页时可以选择随机页来驱逐，但如果选择不常用的页，系统性能会好得多。
如果移除了一个频繁使用的页，它可能很快又需要被调入，从而导致额外开销。

### 最优算法

最优的页置换算法容易描述但无法实际实现。

该算法的唯一问题是无法实现。在缺页发生时，操作系统无法知道每个页将在何时被再次引用。
（我们在之前的[最短作业优先调度算法](/docs/CS/OS/process.md?id=shortest-job-first)中看到过类似情况——系统如何判断哪个作业最短？）
不过，通过在模拟器上运行程序并跟踪所有页引用，可以在第二次运行时利用第一次运行收集的页引用信息来实现最优页置换。

### NRU 算法

NRU（最近未使用）算法从编号最低的非空类中随机移除一个页。
该算法隐含的思想是：移除一个至少在一个时钟滴答（通常约 20 毫秒）内未被引用但被修改的页，比移除一个正在被频繁使用的干净页更好。
NRU 的主要吸引力在于它易于理解、实现效率适中，并且性能虽然肯定不是最优，但可能足够。

### FIFO 算法

### 第二次机会算法

对 FIFO 的一个简单修改，避免丢弃频繁使用的页的问题，是检查最旧页的 R 位。
如果为 0，则该页既旧又未使用，因此立即被替换。
如果 R 位为 1，则清除该位，将该页放到页列表的末尾，并更新其加载时间，就像它刚刚到达内存一样。然后继续搜索。
这种算法的操作称为第二次机会。

### 时钟算法

虽然第二次机会是合理的算法，但它不必要地低效，因为它不断在列表中移动页面。
更好的方法是将所有页帧以时钟形式保存在循环列表中。指针指向最旧的页。

### LRU 算法

对最优算法的一个良好近似基于这样的观察：在最近几条指令中被频繁使用的页很可能很快再次被频繁使用。
相反，长时间未使用的页可能会在很长时间内保持未使用。
这个想法提出了一种可实现的算法：当缺页发生时，丢弃最长时间未被使用的页。
这种策略称为*LRU（最近最少使用）*分页。

虽然 LRU 在理论上是可行的，但它远不便宜。
要完全实现 LRU，需要维护内存中所有页的链表，最近使用的页在表头，最近最少使用的页在表尾。
困难在于链表必须在每次内存引用时更新。
在链表中查找页、删除它、然后将其移动到表头是非常耗时的操作，即使在硬件中也是如此（假设可以构建这样的硬件）。

### 工作集算法

在最纯粹的分页形式中，进程启动时没有任何页在内存中。
一旦 CPU 尝试获取第一条指令，就会发生缺页中断，导致操作系统调入包含第一条指令的页。
全局变量和栈的其他缺页通常会很快跟随。
一段时间后，进程拥有所需的大部分页，并稳定下来以相对较少的缺页运行。
这种策略称为*请求调页*，因为页仅在需要时才被加载，而不是提前加载。

进程当前使用的页集合是其工作集。
如果整个工作集在内存中，进程将运行而不会引起许多缺页，直到进入另一个执行阶段（例如，编译器的下一遍）。
如果可用内存太小而无法容纳整个工作集，进程将引起许多缺页并运行缓慢，
因为执行一条指令只需几纳秒，而从磁盘读取一页通常需要 10 毫秒。以每 10 毫秒一条或两条指令的速度，完成将需要很长时间。
每几条指令就引起一次缺页的程序被称为*抖动*。

因此，许多分页系统尝试跟踪每个进程的工作集，并确保在让进程运行之前其工作集在内存中。这种方法称为*工作集模型*。
它旨在极大地降低缺页率。在让进程运行之前加载页面也称为*预调页*。请注意，工作集会随时间变化。

### WSClock 算法

### 页置换算法总结

| 算法 | 说明 |
| ------------------------------------------------------------ | ------------------------------------------------ |
| 最优 | 不可实现，但作为基准有用 |
| NRU（最近未使用） | 对 LRU 的非常粗略的近似 |
| FIFO（先进先出） | 可能丢弃重要页面 |
| 第二次机会 | 对 FIFO 的重大改进 |
| 时钟 | 现实可行 |
| LRU（最近最少使用） | 优秀，但难以精确实现 |
| NFU（不常用） | 对 LRU 的相当粗略的近似 |
| 老化 | 高效算法，能很好地近似 LRU |
| 工作集 | 实现成本稍高 |
| WSClock | 良好的高效算法 |

总的来说，最好的两种算法是老化和 WSClock。它们分别基于 LRU 和工作集。
两者都提供良好的分页性能，并且可以高效实现。
