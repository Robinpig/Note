## Introduction

哈希是一种数学函数，它将任意长度的输入转换为固定长度的加密输出。
因此，无论原始数据量或文件大小如何，其唯一的哈希值始终大小相同。
此外，哈希不能用于从哈希输出"逆向工程"出输入，因为哈希函数是"单向"的（就像绞肉机；你无法把绞碎的牛肉变回牛排）。
不过，如果你对相同的数据使用这样的函数，其哈希值将是相同的，因此如果你已经知道其哈希值，就可以验证数据是否相同（即未被修改）。

我们讨论哈希表 ADT，它只支持二叉搜索树所允许操作的一个子集。

哈希表的实现通常被称为 hashing。
哈希是一种在常数平均时间内执行插入、删除和查找的技术。
需要元素之间排序信息的树操作无法高效支持。
因此，像 find_min、find_max 以及在线性时间内按排序顺序打印整个表等操作都不被支持。

## General Idea

理想的哈希表数据结构仅仅是一个包含键的固定大小数组。
通常，键是一个带有关联值（例如工资信息）的字符串。
我们将表大小称为 *H_SIZE*，需理解这是哈希数据结构的一部分，而不仅仅是全局变量。
通常约定表从 0 到 *H_SIZE-1*。

每个键被映射到 0 到 *H_SIZE* - *1* 范围内的某个数字，并放入相应的单元格中。
这种映射称为**哈希函数**，理想情况下应该易于计算，并确保任意两个不同的键获得不同的单元格。

由于单元格数量有限，而键的供应几乎是无穷无尽的，这显然是不可能的，因此我们寻求一种能够将键均匀分布在各个单元格中的哈希函数。

这就是哈希的基本思想。
剩下的问题只涉及选择函数、决定当两个键哈希到相同值时的处理方法（这称为**冲突**），以及确定表的大小。

## Hash Function

如果所有键事先已知，可以使用[完美哈希函数](https://en.wikipedia.org/wiki/Perfect_hash_function)来创建没有冲突的完美哈希表。
如果使用最小完美哈希，哈希表中的每个位置也都可以被利用。

完美哈希在所有情况下都允许常数时间的查找。
这与大多数链地址法和开放地址法形成对比，后者的查找时间平均较低，但可能非常大，达到 $O(n)$，例如当所有键哈希到少数几个值时。

### Key statistics

哈希表的一个关键统计量是 `load factor`（负载因子），定义为

$$
loadfactor(\alpha)=\frac{n}{k}
$$

其中

- n 是哈希表中已占用的条目数。
- k 是桶的数量。

哈希表的[性能](https://en.wikipedia.org/wiki/Computer_performance)随着负载因子 $\alpha$ 的增加而恶化。
因此，当负载因子 $\alpha$ 接近 1 时，哈希表会被调整大小或重新哈希。
当负载因子低于 ${ \alpha_{max}/4}$ 时，表也会被调整大小。
可接受的负载因子 $\alpha$ 值包括 0.6 和 0.75。

## Hash Collision

哈希是一种通过重新压缩数据来提高效率的解决方案。
然而，由于哈希函数生成的哈希值有限且数据可能更多，仍然存在不同数据经过哈希函数处理后对应相同值的情况。
这就是出现了**哈希冲突**。

### 哈希冲突的影响因素

负载因子（负载因子 = 总数据量/哈希表长度）、哈希函数、冲突处理方法

### 解决哈希冲突的四种方式

- 开放地址法
- 链地址法
- 再哈希
- 溢出表

### Open Hashing (Separate Chaining)

Separate Chaining（分离链接法）

*open hashing*（开放哈希）或 *separate chaining*（分离链接）是为所有哈希到相同值的元素维护一个列表。

执行*插入*时，我们遍历相应的链表以检查元素是否已存在（如果允许重复，通常保留一个额外字段，匹配时递增该字段）。
如果元素是新元素，则将其插入链表的前端或末端，以最简便的方式为准。
这通常是在编写代码时最容易解决的问题。

> [!NOTE]
>
> 有时新元素会被插入到链表的前端，这不仅方便，而且因为经常发生的情况是，最近插入的元素在不久的将来最有可能被访问。

除了链表之外，任何方案都可以用来解决冲突——二叉搜索树甚至另一个哈希表都可以，
但我们期望如果表很大且哈希函数良好，所有链表都应该很短，因此不值得尝试任何复杂的方法。

我们将哈希表的负载因子 $\delta$ 定义为哈希表中元素数量与表大小的比率。
在上面的例子中，$\delta = 1.0$。链表的平均长度为 $\delta$。
执行搜索所需的工作量是计算哈希函数的常数时间加上遍历链表的时间。

在不成功的搜索中，平均需要遍历 $\delta$ 个链接（不包括最后的 NULL 链接）。
成功的搜索需要遍历约 $1 + (\delta/2)$ 个链接，因为保证必须遍历一个链接（因为搜索成功），并且我们预计还要在链表中走一半路程才能找到匹配项。
这个分析表明表的大小并不真正重要，负载因子才是关键。
开放哈希的一般规则是使表大小大约等于预期的元素数量（换句话说，让 $\delta \approx 1$）。
如前所述，保持表大小为质数以确保良好分布也是一个好主意。

开放哈希的缺点是需要指针。
这会因为分配新单元格所需的时间而略微减慢算法速度，并且本质上需要实现第二个数据结构。

### Closed Hashing (Open Addressing)

与分离链接一样，开放地址法是处理冲突的一种方法。
在开放地址法中，所有元素都存储在哈希表本身中。
因此，在任何时候，表的大小必须大于或等于键的总数（注意，如果需要，我们可以通过复制旧数据来增加表大小）。

- Insert(k)：持续探测直到找到空槽。找到空槽后，插入 k。
- Search(k)：持续探测直到槽的键等于 k 或达到空槽。
- Delete(k)：***删除操作很有趣***。如果直接删除键，搜索可能会失败。因此，已删除键的槽被特殊标记为"deleted"。
  insert 可以将元素插入已删除的槽，但 search 不会在已删除的槽处停止。
- [ThreadLocalMap in Java](/docs/CS/Java/JDK/Concurrency/ThreadLocal.md?id=hash)
- HashMap in Python
- [map - Golang](/docs/CS/Go/struct/map.md)

Closed hashing（封闭哈希），也称为 open addressing（开放地址法），是使用链表解决冲突的另一种方法。
在封闭哈希系统中，如果发生冲突，会尝试其他单元格，直到找到空单元格。

因为所有数据都存储在表内部，所以封闭哈希需要比开放哈希更大的表。
通常，封闭哈希的负载因子应低于 $\delta = 0.5$。

**标准删除不能在封闭哈希表中执行，因为该单元格可能导致冲突绕过它。**
因此，封闭哈希表需要惰性删除，尽管在这种情况下实际上并没有"惰性"的含义。

#### Linear Probing

对于线性探测，让哈希表接近满是一个坏主意，因为性能会急剧下降。

#### Quadratic Probing

对于二次探测，情况更为严峻：一旦表超过半满，甚至在半满之前（如果表大小不是质数），都无法保证找到空单元格。
这是因为最多只有一半的表可以用作解决冲突的替代位置。

## Rehashing

*Rehashing*（再哈希）显然是一个非常昂贵的操作——运行时间为 $O(n)$，因为有 n 个元素需要重新哈希，且表大小约为 2n，但实际上这并不那么糟糕，因为它发生得非常不频繁。

再哈希可以通过几种方式与二次探测一起实现。

- 一种选择是在表半满时立即再哈希。
- 另一种极端是仅在插入失败时再哈希。
- 第三种折中策略是在表达到某个负载因子时再哈希。

由于性能确实随着负载因子的增加而下降，采用良好截止值的第三种策略可能是最好的。

再哈希使程序员不必担心表的大小，并且很重要，因为在复杂程序中哈希表不能任意大。

## Consistent Hash

一致哈希算法旨在解决哈希算法的迁移成本问题。
因此，当使用一致哈希算法进行哈希寻址时，可以通过增加节点数量来减少节点宕机对整个集群的影响，
以及在故障恢复时需要迁移的数据量。
之后，在需要时，可以增加节点数量以提高系统的容灾能力和效率。
在进行数据迁移时，只需迁移部分数据即可实现集群稳定。

我们都知道，常见的哈希算法是对节点数量进行取模路由寻址，同样地，一致哈希也使用取模运算，
但与哈希算法对节点数量取模不同，一致哈希算法对 2^32 取模。
可以想象，一致哈希算法将整个哈希值空间组织成一个虚拟的圆环，即哈希环。

[Consistent Hashing and Random Trees: Distributed Caching Protocols for Relieving Hot Spots on the World Wide Web](https://www.cs.princeton.edu/courses/archive/fall09/cos518/papers/chash.pdf)

虽然一致性 Hash 算法已经十分完善，但是还是有很多不足的地方

1. Hash 环上的节点非常多或者更新频繁时，查询效率比较低下
2. 整个 Hash 环需要一个服务路由来做负载均衡，存在单点问题

一致性 Hash 算法首先需要考虑的问题就是：构造一个长度为 `2^32` 的整数环，根据节点名称的 Hash 值将服务器节点放置在这个 Hash 环上

## Summary

哈希表可以在常数平均时间内实现 *insert* 和 *find* 操作。
在使用哈希表时，特别需要注意负载因子等细节，否则时间界是无效的。
当键不是短字符串或整数时，仔细选择哈希函数也很重要。

对于开放哈希，负载因子应接近 1，尽管除非负载因子变得非常大，否则性能不会显著下降。
对于封闭哈希，负载因子不应超过 0.5，除非完全不可避免。
如果使用线性探测，性能会随着负载因子接近 1 而迅速恶化。
可以实现再哈希来允许表增长（和收缩），从而维持合理的负载因子。
如果空间紧张且无法声明一个巨大的哈希表，这一点很重要。

二叉搜索树也可以用来实现 *insert* 和 *find* 操作。
虽然得到的平均时间界是 $O(\log{n})$，但二叉搜索树也支持需要排序的例程，因此更加强大。
使用哈希表，无法找到最小元素。除非知道确切的字符串，否则无法高效搜索字符串。
二叉搜索树可以快速找到某个范围内的所有元素；哈希表不支持这一点。
此外，$O(\log{n})$ 的界不一定比 $O(1)$ 大很多，特别是因为搜索树不需要乘法或除法。

另一方面，哈希的最坏情况通常源于实现错误，而排序的输入可能导致二叉树性能不佳。
平衡搜索树的实现相当昂贵，因此如果不需要排序信息，并且怀疑输入可能是排序的，那么哈希是首选的数据结构。

哈希的应用非常丰富。编译器使用哈希表来跟踪源代码中声明的变量。
这种数据结构称为**符号表**。
哈希表是解决这个问题的理想应用，因为只需要执行 *insert* 和 *find*。
标识符通常很短，因此哈希函数可以快速计算。

对于节点具有实际名称而不是编号的图论问题，哈希表非常有用。
在这里，随着输入的读取，顶点按出现顺序从 1 开始分配整数。
同样，输入很可能有大量按字母顺序排列的条目。
例如，顶点可能是计算机。
那么，如果某个特定的安装将其计算机列为 *ibm1, ibm2, ibm3*...，使用搜索树可能会对效率产生显著影响。

哈希表的第三个常见用途是在游戏程序中。
当程序搜索不同的走法时，它通过基于位置计算哈希函数（并存储该位置的走法）来跟踪已见过的位置。
如果同一个位置再次出现（通常是通过简单的走法转置），程序可以避免昂贵的重新计算。
所有游戏程序的这个通用特性称为置换表。

哈希的另一个用途是在线拼写检查器。
如果拼写错误检测（而非纠正）很重要，可以预先哈希整个字典，并在常数时间内检查单词。
哈希表非常适合这个用途，因为按字母顺序排列单词并不重要；按单词在文档中出现的顺序打印拼写错误是完全可接受的。
