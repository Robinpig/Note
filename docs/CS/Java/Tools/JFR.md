## Introduction

Java Flight Recorder（JFR）是一种用于收集运行中 Java 应用程序的诊断和分析数据的工具。它集成到 Java 虚拟机（JVM）中，几乎不会造成性能开销，因此即使在负载较重的生产环境中也可以使用。使用默认设置时，内部测试和客户反馈都表明性能影响小于百分之一。对于某些应用程序，它可能更低。然而，对于短期运行的应用程序（不属于生产环境中运行的应用程序类型），相对启动和预热时间可能更长，这可能会使性能影响超过百分之一。JFR 收集关于 JVM 以及在其上运行的 Java 应用程序的数据。

与其他类似工具相比，JFR 具有以下优势：
- 提供更好的数据：JFR 使用的连贯数据模型使得交叉引用和过滤事件更加容易。
- 允许第三方事件提供者：一组 API 允许 JFR 监控第三方应用程序，包括 WebLogic Server 和其他 Oracle 产品。
- 降低总体拥有成本：JFR 使您能够花更少的时间诊断和排查问题，降低运营成本和业务中断，在问题发生时提供更快的解决时间，并提高系统效率。

JFR 主要用于：
- Profiling（性能分析）
  JFR 持续保存大量关于运行系统的数据。此 profiling 信息包括线程样本（显示程序花费时间的位置）、锁 profiles 和垃圾回收详细信息。
- Black Box Analysis（黑盒分析）
  JFR 持续将信息保存到循环缓冲区。当检测到异常时可以访问此信息以查找原因。
- Support and Debugging（支持和调试）
  JFR 收集的数据在联系 Oracle 支持以帮助诊断 Java 应用程序问题时可能至关重要。

JFR有两种主要的概念：事件和数据流

JFR 在 Java 应用程序运行时收集 JVM 中发生的事件。这些事件与 JVM 自身的状态或程序的状态相关。事件具有名称、时间戳和附加信息（如线程信息、执行栈和堆状态）。

JFR 收集三种类型的事件：
- instant event（即时事件）一旦发生就立即记录
- duration event（持续时间事件）如果其持续时间超过指定阈值则记录
- sample event（采样事件）用于对系统活动进行采样

JFR 收集的事件包含大量数据。因此，在设计上，JFR 足够快，不会妨碍程序。
JFR 将事件数据保存在单个输出文件 flight.jfr 中。

众所周知，磁盘 I/O 操作相当昂贵。因此，JFR 使用各种缓冲区来存储收集的数据，然后将数据块刷新到磁盘。事情可能变得稍微复杂，因为在同一时刻，程序可能有多个具有不同选项的注册进程。

因此，我们可能会在输出文件中发现比请求更多的数据，或者数据可能不是按时间顺序排列的。如果我们使用 JMC，可能甚至不会注意到这一点，因为它按时间顺序可视化事件。

在极少数情况下，JFR 可能无法刷新数据（例如，当事件太多或发生断电时）。如果发生这种情况，JFR 会尝试通知我们输出文件可能缺少一部分数据。

在 JVM 启动文件的路径中添加 -XX:StartFlightRecording=duration,filename=jfr_recording.jfr 参数，其中 duration 表示录制时长，单位为毫秒，filename 表示录制文件的名称

JFR本身是基于周期性对JVM虚拟机的运行状况进行采样记录的，其采样的频率可以通过其参数传入;只是需要留意的是，采样间隔越小对系统的性能干扰就越大。 和传统的JProfiler/VisualVM这些基于JMX的工具所不同的是，JFR记录的信息是近似而非精确的；当然大部分情况下这些模糊性信息就足够说明问题了

需要注意的是，JFR 录制过程中会对应用程序的性能产生一定的影响，因此需要在测试环境中进行测试，以确保不会影响生产环境中的应用程序性能。
1. JFR 的线程堆栈 dump 采集默认周期是everyChunk，也就是每次新建一个Chunk就会采集一次。默认每个Chunk大小为 12M，在线程比较多的时候，堆栈 dump 可能大于 12M，导致 JFR 一直切换新的 Chunk，然后又触发evenryChunk导致无限循环而雪崩。 
2. 对于 JFR 默认事件采集配置（位于JDK目录/lib/jfr/default.jfc），每个采集周期和Chunk相关的，都要谨慎处理，最好周期通过固定时间写死，例如每一分钟等等，不要使用Chunk作为指标，防止上面这个问题。
