## Introduction

ReadWriteLock 维护一对关联的锁，一个用于只读操作，一个用于写入。读锁可以由多个读线程同时持有，只要没有写线程。写锁是独占的。

所有 ReadWriteLock 实现必须保证 writeLock 操作的内存同步效应（如 Lock 接口中指定的）也适用于关联的 readLock。也就是说，成功获取读锁的线程将看到先前释放写锁时所做的所有更新。
读写锁允许比互斥锁更高程度的并发性访问共享数据。它利用了这样的特性：虽然一次只有一个线程（写线程）可以修改共享数据，但在许多情况下任意数量的线程可以同时读取数据（因此是读线程）。理论上，使用读写锁所允许的并发性增加将比使用互斥锁带来性能改进。实际上，这种并发性的增加只会在多处理器上完全实现，并且只有在共享数据的访问模式合适时才成立。

读写锁是否能比互斥锁提高性能取决于读取频率与修改频率的比较、读写操作的持续时间以及数据的争用程度——即同时尝试读取或写入数据的线程数量。例如，一个初始填充数据后不经常修改但频繁搜索的集合（如某种目录）是使用读写锁的理想候选。然而，如果更新变得频繁，那么数据大部分时间被独占锁定，并发性几乎没有提高。此外，如果读操作过于短暂，读写锁实现的开销（其本质上比互斥锁更复杂）可能主导执行成本，特别是因为许多读写锁实现仍然通过一小段代码序列化所有线程。最终，只有性能分析和测量才能确定使用读写锁是否适合你的应用。

虽然读写锁的基本操作很简单，但实现必须做出许多策略决策，这些决策可能会影响读写锁在给定应用中的有效性。这些策略包括：

1. 当写者释放写锁时，确定在读者和写者都在等待的情况下是授予读锁还是写锁。写者优先是常见的，因为期望写入是短暂且不频繁的。读者优先较不常见，因为如果读者频繁且存活时间长，可能导致写操作的长时间延迟。公平或"按顺序"的实现也是可能的。
2. 确定当读者活跃且写者等待时，请求读锁的读者是否被授予读锁。优先考虑读者可能无限期延迟写者，而优先考虑写者可能降低并发潜力。
3. 确定锁是否可重入：持有写锁的线程能否重新获取它？持有写锁时能否获取读锁？读锁本身是否可重入？
4. 写锁能否降级为读锁而不允许中间写者介入？读锁能否升级为写锁，优先于其他等待的读者或写者？

在评估给定实现对你的应用的适用性时，应考虑所有这些事项。

```java
public interface ReadWriteLock {
    Lock readLock();
    Lock writeLock();
}
```
